⚙️ Knowledge: IT & System

PrintBuddy System-Architektur

Übersicht: PrintBuddy ist eine Django-Applikation, die auf einem Hetzner VPS läuft und mit einem NAS-Homeserver (Synology) via Tailscale VPN verbunden ist. GPU-Workloads laufen auf RunPod (Cloud GPU), CPU-Tasks lokal auf dem VPS.

Daten-Flow: Studio (Browser) → VPS (Django) → Celery → RunPod (GPU) → NAS (Storage) → Nginx (Delivery)

🏗️ Komponenten-Übersicht
🌐 Hetzner VPS
IP: 67.86.108.37
OS: Ubuntu 22.04 LTS
CPU: 4 vCores
RAM: 8 GB
Rolle: Django, PostgreSQL, Redis, Celery CPU-Worker, Nginx
💾 Synology NAS
Modell: DS925+
Mount: /mnt/agency_nas
Protokoll: NFSv4 via Tailscale
Rolle: Langzeit-Storage, Druckdatei-Bundles, Backups
🚀 RunPod (GPU)
Region: EU (Niederlande)
GPU: On-Demand (A4000/A5000)
Kosten: ~€0.03–0.04 pro Bild
Rolle: FLUX/SDXL Generierung, Upscaling
🔐 Tailscale VPN
Network: 100.x.x.x (privat)
Verbindung: VPS ↔ NAS
Rolle: Sichere NFS-Verbindung ohne öffentliche Ports

Warum diese Stack-Wahl?

VPS statt eigener Server: Niedrige Fixkosten, einfaches Backup, professionelle Uptime.
RunPod statt eigene GPU: Nur zahlen wenn generiert wird. Keine Hardware-Wartung.
NAS via Tailscale: Günstig (einmalig €500 statt Cloud-Abo), hohe Kapazität, volle Kontrolle.
PostgreSQL lokal: Niedrige Latenz, kein externes DB-Abo nötig.

🔐 Tailscale VPN — Netzwerk & Konten

Was ist Tailscale?

Zweck: Sicheres, verschlüsseltes VPN zwischen VPS und NAS.
Vorteil: Keine öffentlichen NFS-Ports nötig. WireGuard-basiert (schnell, sicher).
Network: 100.x.x.x (privates Mesh-Netzwerk)

Registrierte Geräte im Tailscale-Netz

1. Hetzner VPS (printbuddy-vps)
• Tailscale IP: 100.x.x.x (variabel)
• Hostname: printbuddy-vps
• Rolle: Client (mountet NAS via NFS)

2. Synology NAS (printbuddy-nas)
• Tailscale IP: 100.y.y.y (variabel)
• Hostname: printbuddy-nas
• Rolle: Server (expose NFSv4 im Tailscale-Netz)

3. Lokaler Dev-Rechner (optional)
• Für Admins: Direkter SSH-Zugang zum VPS via Tailscale
• Für NAS-Maintenance: Synology DSM Web-UI via Tailscale IP

NFS Mount-Konfiguration

VPS /etc/fstab:
100.y.y.y:/volume1/agency_nas /mnt/agency_nas nfs4 defaults,_netdev 0 0

NAS NFS-Freigabe (Synology DSM):
• Shared Folder: agency_nas
• NFS-Berechtigung: nur 100.x.x.x (VPS Tailscale IP)
• Squash: No root squash (UID=1024 mapped zu printbuddy-nas User)
🚨 Sicherheit:
NAS ist NICHT öffentlich erreichbar!
Nur über Tailscale VPN zugänglich.
Keine Ports (80/443/22/2049) im Router forwarded.
🛠️ Wo was ändern? — Entwickler-Guides

🎨 Frontend (Studio UI) ändern

Dateien: templates/studio/*.html
Styling: Inline-CSS in templates/studio/base.html (<style>)
Neustart nötig? Nein (DEBUG=True), Ja (Produktion → collectstatic)

Beispiel: Navigation erweitern
templates/studio/base.html → <nav> Block
Link hinzufügen: <a href="{% url 'studio:neue_view' %}">Label</a>

⚙️ Backend (Django Logic) ändern

Views: studio/views.py, shop/views.py etc.
Models: jobs/models.py, gallery/models.py etc.
Services: jobs/services.py (komplexe Business-Logik)

Nach Änderungen:
1. python manage.py makemigrations (bei Model-Änderungen)
2. python manage.py migrate
3. sudo systemctl restart gunicorn (Produktion)
4. Bei Celery-Tasks: sudo systemctl restart celery-gpu celery-cpu

🚀 GPU-Pipeline ändern

Tasks: gpu/tasks.py (generate_image, upscale_image)
RunPod Config: Env-Vars in .env (RUNPOD_API_KEY, RUNPOD_ENDPOINT_ID)

Beispiel: Neues Modell hinzufügen
1. RunPod Endpoint erstellen mit neuem Modell
2. jobs/models.py → PipelineTemplate.default_model Choices erweitern
3. gpu/tasks.py → generate_image Model-Handling anpassen
4. Migration + Celery-Restart

💾 NAS Ordnerstruktur ändern

Ort: Synology DSM → File Station → agency_nas
Wichtig: Permissions beachten! chmod 755 für Ordner, 644 für Dateien.

Nach Struktur-Änderung:
• settings.py → NAS_BASE_PATH anpassen (falls Root ändert)
• Tasks in gpu/ und postprocess/ überprüfen (hardcoded Pfade?)
• copilot-instructions.md → Section 6 aktualisieren

🌐 VPS Server-Konfiguration

Nginx: /etc/nginx/sites-available/printbuddy
Gunicorn: /etc/systemd/system/gunicorn-printbuddy.service
Celery: /etc/systemd/system/celery-printbuddy-gpu.service + cpu.service
PostgreSQL: localhost:5432 (nur lokal erreichbar)
Redis: localhost:6379 (nur lokal erreichbar)

Nach Config-Änderung:
• sudo systemctl daemon-reload (bei systemd-Änderungen)
• sudo systemctl restart SERVICE_NAME
• sudo nginx -t && sudo systemctl reload nginx (Nginx-Config-Test)
⚠️ WICHTIG — Wirkung von Änderungen:
• Frontend (HTML/CSS): Nur Browser-Refresh (dev), collectstatic (prod)
• Backend (Views/Models): Gunicorn-Restart (prod)
• Celery-Tasks: Celery-Workers-Restart (gpu+cpu)
• Systemd-Services: daemon-reload + Service-Restart
• NAS-Struktur: Kein Restart, aber Code anpassen (Pfade!)

⚖️ Tech-Stack: Was & Warum — Upgrade-Szenarien

Django 4.2 LTS

Gewählt weil:
• Mature, stabil, große Community
• ORM + Admin-UI out-of-the-box
• Gut für monolithische Architektur (kein Microservices-Overhead)

Alternative wäre gewesen: FastAPI (Async, API-first) → aber mehr manueller Setup-Aufwand

Upgrade-Szenario:
Django 5.x → LTS-Pfad weiter (2026: Django 5.2 LTS).
Nur upgraden wenn kritische Security-Fixes oder neue Features benötigt.

PostgreSQL 15

Gewählt weil:
• Open Source, DSGVO-konform (lokale Kontrolle)
• JSON-Support für flexible Felder (JSONField)
• Deutlich performanter als MySQL für Django ORM

Alternative wäre gewesen: MySQL → aber schlechtere Django-Integration, keine JSON-Aggregates

Upgrade-Szenario:
PostgreSQL 16/17 → Minor-Upgrades safe. Major-Version-Upgrade nur wenn neue Features kritisch (z.B. Performance).

Celery 5 + Redis 7

Gewählt weil:
• Best Practice für asynchrone Tasks in Django
• Redis schneller als RabbitMQ für kleine/mittlere Workloads
• Einfaches Monitoring (Flower)

Alternative wäre gewesen: Django-Q oder Huey → aber weniger Community-Support

Upgrade-Szenario:
Redis Sentinel/Cluster → Nur wenn HA (High Availability) kritisch wird (>10k Jobs/Tag).
RabbitMQ → Nur wenn komplexere Task-Routing-Logik benötigt.

Hetzner VPS

Gewählt weil:
• DSGVO-konform (DE Datacenter)
• Günstig (~€10–20/Monat für 4 vCores, 8 GB RAM)
• Root-Zugang (volle Kontrolle)

Alternative wäre gewesen: AWS/GCP/Azure → aber deutlich teurer, Vendor Lock-in

Upgrade-Szenario:
Dedicated Server (Hetzner AX-Serie) → Wenn VPS-CPU-Limit erreicht (>500 Orders/Tag).
Kubernetes → Nur wenn Multi-Region-Deployment kritisch wird.

RunPod On-Demand GPU

Gewählt weil:
• Nur zahlen wenn generiert wird (~€0.03–0.04 pro Bild)
• Keine Wartezeiten bei Spot-Instanzen
• EU-Region verfügbar (Niederlande)

Alternative wäre gewesen: Eigener GPU-Server (Hetzner GX2-15, ~€107/Mo) → break-even bei ~200 Jobs/Monat

Upgrade-Szenario:
Hetzner GPU-Dedicated → Wenn regelmäßig >300 Bilder/Monat generiert werden (Kosten-Break-even).
Vast.ai Fallback bleibt als Backup.

NAS via Tailscale

Gewählt weil:
• Einmalig €500 statt Cloud-Abo (€20+/Monat)
• Volle Kontrolle, hohe Kapazität (4+ TB)
• Tailscale = sicher ohne öffentliche Ports

Alternative wäre gewesen: S3 (AWS) oder Hetzner Storage Box → aber laufende Kosten, kein lokaler Zugriff

Upgrade-Szenario:
S3-Migration → Nur wenn NAS-Heimanschluss zum Problem wird (z.B. Umzug, Downtime).
Hybrid: NAS primär, S3 als Backup (z.B. via rclone sync).
💡 Upgrade-Philosophie:
"Don't fix what isn't broken."
Upgrades nur wenn:
1. Security-Patches kritisch
2. Performance wird zum Bottleneck (messbar!)
3. Neue Features business-kritisch

Kein Upgrade nur wegen "neueste Version".
👤 Konten & Zugänge — Wer hat wo Zugriff?

Django User-Rollen

1. Admins (is_staff=True + is_superuser=True)
• Voll-Zugriff auf Django Admin
• Können Jobs starten (ADR-11)
• Können GalleryImage is_public setzen
• Zugriff auf alle Studio-Features

2. Studio Workers (Gruppe: studio_workers)
• Können Jobs erstellen (nur Draft)
• Können Bilder für Galerie vormerken
• Können Prompt-Bibliothek nutzen
• Können Jobs/Results sehen (nur eigene)
• KEIN Admin-Zugriff
• KÖNNEN NICHT Jobs starten (ADR-11)

3. Druckpartner (Gruppe: print_partners)
• Können eigene Bestellungen sehen
• Können Bundles herunterladen
• Können Tracking-Codes eintragen
• KEIN Zugriff auf Studio oder Admin

VPS Linux-User

1. root
• System-Administration
• SSH-Zugang für Admins
• Nginx/PostgreSQL/Redis-Management

2. www-data
• Nginx Worker-Prozess
• Liest NAS-Dateien (/media/)
• KEIN Schreibzugriff auf NAS

3. printbuddy (App-User)
• Gunicorn + Celery laufen als dieser User
• Schreibzugriff auf NAS via UID=1024-Mapping
• Home: /opt/printbuddy/

NAS Synology-User

1. admin (Synology DSM Admin)
• Full Admin-Zugriff auf Synology DSM Web-UI
• Kann NFS-Freigaben konfigurieren
• Kann Tailscale-Einstellungen ändern

2. printbuddy-nas (UID=1024)
• Mapped zu VPS printbuddy User
• Schreibrechte auf agency_nas Shared Folder
• Kein DSM-Login (nur für NFS-Mapping)
